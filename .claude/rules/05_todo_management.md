# TODO管理ルール

## 🎯 基本原則

### TODO管理の重要性
- **TODOリストは開発の羅針盤**: 何を実装すべきか、何が完了したかを明確にする
- **進捗の可視化**: ユーザーとClaude両方が進捗状況を把握できる
- **タスクの抜け漏れ防止**: すべての機能が確実に実装される
- **品質保証**: テストやレビューもTODOに含まれているため、品質が担保される

---

## 📋 TODOリストの構成

### Phase別TODO
このプロジェクトには3つのPhase別TODOリストがあります：

1. **PHASE1_HIGH_PRIORITY.md** (優先度: 高)
   - 基本的なSNS機能（テキスト投稿のみ）
   - ユーザー認証、投稿、コメント、いいね、フォロー、通知、通報

2. **PHASE2_MEDIUM_PRIORITY_PART1.md** (優先度: 中)
   - Firebase Storage統合
   - 画像/動画アップロード、リツイート、ハッシュタグ、検索、ブロック

3. **PHASE3_MEDIUM_PRIORITY_PART2.md** (優先度: 中)
   - 管理者機能、アナリティクス

### TODOの構造
```markdown
## 1. カテゴリ

### 1.1 サブカテゴリ
- [ ] 未完了のタスク
- [x] 完了したタスク
  - [ ] サブタスク1
  - [ ] サブタスク2
```

## ✅ TODO更新の必須タイミング

### 1. コード変更前
**必ず該当するTODOを確認する**
- どの機能を実装するのか
- 実装すべき内容は何か
- 関連する仕様書はどれか

例：
```
投稿作成機能を実装する前に：
1. docs/todo/PHASE1_HIGH_PRIORITY.md を開く
2. 「2.7 投稿機能 (Posts)」セクションを確認
3. Repository, Service, DTO, Handler の実装項目を確認
4. docs/specs/02_DATABASE_SCHEMA.md でデータベース設計を確認
5. docs/specs/03_API_SPECIFICATION.md でAPI仕様を確認
```

### 2. タスク完了時
**必ずチェックマークを入れる**
```markdown
# 変更前
- [ ] 投稿作成Repository実装

# 変更後
- [x] 投稿作成Repository実装
```

### 3. 問題発生時
**TODOに問題を記録する**
```markdown
- [ ] 投稿作成API テスト
  - ⚠️ 問題: 画像アップロード時にタイムアウトエラー
  - 原因調査中: Firebase Storageの設定を確認
```

### 4. 新しいタスクが発生した場合
**TODOに追加する**
```markdown
- [x] 投稿作成機能実装
- [ ] 投稿作成時のバリデーション追加（追加タスク）
  - 理由: テスト中にバリデーション不足を発見
```

## 📝 TODOの更新方法

### ファイルを直接編集
```bash
# TODOファイルを開く
code docs/todo/PHASE1_HIGH_PRIORITY.md

# 完了したタスクにチェックを入れる
- [x] 完了したタスク

# 保存して確認
git diff docs/todo/PHASE1_HIGH_PRIORITY.md
```

### 確認すべきポイント
- [ ] チェックマークは正しく入っているか
- [ ] 新しいタスクは適切な場所に追加されているか
- [ ] 問題や注意事項は記録されているか

## 🔄 開発フロー（TODO中心）

### ステップ1: タスク選択
```
1. 現在のPhaseのTODOリストを開く
2. 次に実装すべきタスクを確認
3. 関連する仕様書を確認
```

### ステップ2: 実装
```
1. タスクを「in_progress」または コメントで進行中とマーク
2. 仕様書に従って実装
3. コーディング規約を遵守
```

### ステップ3: 完了
```
1. 実装が完了したらテスト実施
2. テストがパスしたらTODOにチェック
3. 次のタスクへ
```

### ステップ4: レビュー
```
1. 一つの機能が完了したら統合テスト
2. 問題があればTODOに記録
3. すべて完了したらPhase完了
```

## ⚠️ よくある間違い

### ❌ 間違い1: TODOを確認せずに実装
```
問題: 仕様を見落とす、重複実装、実装漏れ
対策: 必ず実装前にTODOと仕様書を確認
```

### ❌ 間違い2: TODOを更新しない
```
問題: 進捗が不明、何が完了したか分からない
対策: タスク完了時は必ずチェックを入れる
```

### ❌ 間違い3: 問題を記録しない
```
問題: 同じエラーを繰り返す、問題解決が遅れる
対策: 問題発生時はTODOに詳細を記録
```

### ❌ 間違い4: テストをスキップ
```
問題: バグが本番で発覚、品質低下
対策: TODOにはテスト項目も含まれている、必ず実施
```

## ✨ ベストプラクティス

### 1. 毎日の開始時
```
□ 前日のTODOを確認
□ 今日実装するタスクを決定
□ 関連する仕様書を再確認
```

### 2. 実装中
```
□ 小さいタスクごとにチェックを入れる
□ 問題があればすぐに記録
□ コミットメッセージにTODO項目を参照
```

### 3. 終了時
```
□ 今日完了したTODOを確認
□ 明日のタスクを確認
□ 問題や懸念事項を記録
```

### 4. Phase完了時
```
□ すべてのTODOが完了しているか確認
□ 統合テストがすべてパスしているか確認
□ ドキュメントが更新されているか確認
□ 次のPhaseの準備
```

## 📝 コミットメッセージとTODOの連携

### コミットメッセージの例
```
feat: implement user registration endpoint

✅ TODO: PHASE1 - 2.5 認証機能
- POST /api/v1/auth/register 実装
- パスワードのbcryptハッシュ化
- JWT生成機能追加

Ref: docs/todo/PHASE1_HIGH_PRIORITY.md#2.5
```

### 利点
- コミット履歴からTODOの進捗が追跡できる
- 後からどのTODO項目に対応したコミットか分かる
- レビュー時に仕様との整合性を確認しやすい

## 🎓 チェックリスト

開発中、常に以下を意識しましょう：

### 実装前
- [ ] TODOリストで次のタスクを確認した
- [ ] 関連する仕様書を確認した
- [ ] 実装内容を理解した

### 実装中
- [ ] コーディング規約を遵守している
- [ ] 仕様書通りに実装している
- [ ] エラーハンドリングを実装している

### 実装後
- [ ] テストを実施した
- [ ] テストがパスした
- [ ] TODOにチェックを入れた
- [ ] コミットメッセージを記載した

### Phase完了時
- [ ] すべてのTODOが完了している
- [ ] 統合テストがすべてパスしている
- [ ] ドキュメントが最新である
- [ ] 次のPhaseの準備ができている

## 📞 困ったときは

### TODOリストが見つからない
```
docs/todo/ ディレクトリを確認:
- PHASE1_HIGH_PRIORITY.md
- PHASE2_MEDIUM_PRIORITY_PART1.md
- PHASE3_MEDIUM_PRIORITY_PART2.md
```

### どのTODOから始めればいいか分からない
```
1. PHASE1から順番に実装
2. 各Phaseの最初の項目から開始
3. 不明な場合はユーザーに質問
```

### TODOと実際の実装に齟齬がある
```
1. 仕様書を確認
2. 仕様書が正しい場合はTODOを更新
3. 不明な場合はユーザーに確認
```

## まとめ

**TODO管理は開発の成功に不可欠です。**

- ✅ **実装前**: TODOと仕様書を確認
- ✅ **実装中**: 進捗に応じてチェック
- ✅ **実装後**: 完了したTODOをマーク
- ✅ **問題発生時**: TODOに記録

**常にTODOリストを最新の状態に保ち、計画的に開発を進めましょう！**
